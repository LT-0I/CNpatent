# 检索方法论 —— cnpatent-noveltycheck

> 基于 CNIPA 审查实务、WIPO 训练材料、专业检索师指南。Screener 和 Guide agent 的检索策略都基于本文件。

## 1. 关键词块策略（block strategy）

每个核心技术特征建立一个"块"（block）。块内用 `OR` 扩展同义词和异名，块间用 `AND` 连接。这是专业检索师的标准方法。

### 扩展维度

对每个特征，从以下六个维度做同义词扩展：

| 维度 | 说明 | 示例（"神经网络"） |
|---|---|---|
| **中英双语** | 中文术语 + 英文术语 | 神经网络 / neural network |
| **同义词** | 同一概念的不同叫法 | 神经网络 / 人工神经网络 / ANN |
| **上下位** | 上位概念 + 下位概念 | 深度学习 / 神经网络 / CNN |
| **全称缩写** | 完整术语 + 缩写 | CNN / convolutional neural network |
| **异名别称** | 领域内的不同称呼 | MLP / feedforward NN / 前馈网络 |
| **繁简异体** | 中文繁简 / 日韩汉字 | 神经网路 (繁) |

### 示例：UAV 集群专利的关键词块

```
块 1 (主体):
  (无人机 OR UAV OR drone OR "unmanned aerial" 
   OR "unmanned aircraft" OR 多旋翼 OR 四旋翼 
   OR quadrotor OR quadcopter OR 垂直起降)

块 2 (集合):
  (集群 OR swarm OR "multi-UAV" OR "multi-agent" 
   OR formation OR 编队 OR 群体 OR cluster 
   OR 分布式 OR distributed)

块 3 (任务):
  (覆盖 OR coverage OR 侦察 OR reconnaissance 
   OR surveillance OR 巡检 OR patrol OR inspection 
   OR 监控 OR monitoring)

块 4 (核心算法):
  (强化学习 OR "reinforcement learning" OR RL 
   OR Q-learning OR "deep Q-network" OR DQN 
   OR "policy gradient" OR PPO OR 决策)
```

组合方式：`块1 AND 块2 AND 块3 AND 块4`

### 块的数量经验值

- **最少** 3 组（低于 3 组会过度宽泛，噪音太多）
- **推荐** 4-5 组（平衡召回率和准确率）
- **上限** 6-7 组（再多会过度狭窄，漏召回）

### 块的来源

关键词应该从以下来源反向收集，**不要凭空想**：

1. **参考文献的标题 + 关键词 + 摘要**
2. **参考文献引用的同领域文献的术语**
3. **参考文献 Related Work 节里的对比方法名**
4. **第一轮粗检后命中文件的 IPC 里的关键词**
5. **领域综述论文里的术语标准化**
6. **英文维基百科对应条目的 "See also" 和 "Alternative names"**

**迭代**：先用最初的 3 组跑一轮粗检，从 Top 20 命中里提取新的同义词，再跑一轮精检。

## 2. IPC / CPC 分类号策略

### IPC 基本知识

- **IPC**：WIPO 官方分类，年度更新，约 7 万类
- **CPC**：EPO + USPTO 联合分类，月度更新，约 25 万类，比 IPC 更细
- **层级**：部（A-H）→ 大类 → 小类 → 大组 → 小组
- **示例**：`G06N 3/08` 表示 G 部 → 06 大类 (计算;推算) → N 小类 (基于特定计算模型的计算机系统) → 3 大组 (使用生物学模型的计算系统) → 08 小组 (学习方法)

1. **初检用关键词** → 读 Top 20 → 统计高频 IPC
2. **定向检索** 用这些 IPC → 精检
3. **避免** 只用一个主 IPC，应同时使用 **2-4 个** 相关小组
4. **CPC 比 IPC 更细**，有条件优先用 CPC

## 3. Boolean 与近似运算

### 优先级

默认从高到低：

```
NOT > AND > NEAR / WITH > OR
```

用括号 `()` 改变优先级。

### 常用算子（不同数据库语法略有差异）

| 算子 | 含义 | incoPat | Google Patents | CNIPA |
|---|---|:---:|:---:|:---:|
| `AND` | 逻辑与 | ✓ | ✓ | ✓ |
| `OR` | 逻辑或 | ✓ | ✓ | ✓ |
| `NOT` | 逻辑非 | ✓ | `-` | ✓ |
| `"..."` | 短语 | ✓ | ✓ | ✓ |
| `*` | 多字符通配 | ✓ | — | ✓ |
| `?` | 单字符通配 | ✓ | — | ✓ |
| `NEAR/n` | 相距 ≤ n | ✓ | — | ✓ |
| `WITH` | 同一字段 | ✓ | — | ✓ |

### 检索式示例

```
# 基础式
TIABC=(neural network OR 神经网络) 
AND TIABC=(convolution* OR 卷积) 
AND IPC=(G06N OR G06V)
AND AD<=2026-04-15

# 使用近似
TIABC=("reinforcement learning" NEAR/5 swarm)
AND IPC=(G05D OR B64U)

# 使用通配
TIABC=(neur* NEAR/3 network)
AND IPC=(G06N 3/*)   # 注意: 这里的 /* 是 IPC 子组通配
```

### 调试技巧

- **检出太少** → 减少 AND 数量 / 扩展 OR 组
- **检出太多** → 加 IPC 限定 / 加时间限定 / 加语言限定
- **噪音太大** → 用 TIABC 代替全文字段 / 用 `NEAR` 代替 `AND`
- **漏召回** → 检查同义词块是否完整 / 增加 CPC 次分类

## 4. 特征拆解（Claim decomposition）

对本方案的权利要求 1 做逐句拆分：

1. **前序部分**（preamble）：通常是"一种 XX 方法，包括..."，作为领域定位
2. **特征部分**：每个"其特征在于..."或"包括..."后面的技术步骤

### 分类

把每个特征标记为：

| 类别 | 标记 | 说明 |
|---|:---:|---|
| 核心创新点 | ⭐ | 本发明的关键差异 |
| 必要支撑特征 | 🔧 | 实现核心创新的必需组件 |
| 常规实施细节 | ⚪ | 领域内常见做法 |

**检索时**，优先对⭐特征建块，🔧作为辅助验证，⚪不需要检索。

### 示例

```
权利要求 1:
  一种基于强化学习的无人机集群区域覆盖方法, 包括:
  (1) 部署无人机集群, 采集区域信息 [⚪]
  (2) 构建扇区分配强化学习模型 [🔧]
  (3) 采用动态权重调整优先级 [⭐] ← 核心创新
  (4) 各无人机按权重执行局部路径规划 [🔧]
  (5) 通过 LoRa 通信汇总结果 [⚪]

检索策略:
  重点建块: 
    - (3) 的动态权重
    - (2) 强化学习 + 扇区分配
  参考建块:
    - (4) 局部路径规划
  不检索:
    - (1)(5) 是领域常识
```

### 特征对比表

命中文件找到后，建立对比表：

| 本方案特征 | 对比文件 | 段落 | 是否公开 | 差异 |
|---|---|---|:---:|---|
| 特征 ⭐(3) 动态权重 | CN1234567A | 0023 | 否 | 区别特征 |
| 特征 🔧(2) 扇区分配 | CN1234567A | 0034 | 是 | 完全相同 |
| 特征 🔧(4) 路径规划 | CN1234567A | 0045 | 部分 | 用 A*, 本方案用 Dijkstra |

**结论**：如果⭐(3)标"否"，本方案具备新颖性（即使其他都对应）。

## 5. 日期策略

### 申请日 vs 优先权日

- **申请日**：向 CNIPA 提交的日期
- **优先权日**：最早同族申请的日期（Paris Convention 12 个月内）

检索现有技术的截止日应该是：

- **有优先权** → 最早优先权日
- **无优先权** → 申请日

对**尚未申请**的方案（本 skill 的场景），截止日设为**今天**或**预期申请日**。

### 抵触申请的特殊日期规则

- **申请日** < 本方案申请日
- **公开日** ≥ 本方案申请日

这意味着在检索时可能还看不到（因为它还没公开）。只能通过商业库的"早期申请"或"未公开申请"字段查。

### 检索式里的日期语法

| 数据库 | 语法 | 示例 |
|---|---|---|
| incoPat | `AD<=yyyy-mm-dd` | `AD<=2026-04-15` |
| Google Patents | `before:yyyy-mm-dd` | `before:2026-04-15` |
| CNIPA | `申请日<=YYYYMMDD` | `申请日<=20260415` |

## 6. 家族、申请人、发明人过滤

### 同族合并

**必做**。同一发明在多国重复申请会在检索时重复出现。合并后按家族去重。

- **简单家族 (DOCDB)**：同一优先权日的所有申请
- **INPADOC 扩展家族**：任一优先权日相连的申请
- **PatSnap 自建家族**：按主题聚合

默认用简单家族。

### 申请人过滤

**初筛时避免**，除非明确在查某公司的专利布局。用申请人过滤会漏掉大量相关专利。

### 发明人过滤

**仅用于追溯特定研究团队**。比如追溯某位论文作者的专利。

## 7. 迭代与停止准则

### 典型流程

```
1. 粗检 (查全优先, 宽关键词块)
   ↓
2. 读 Top 50 标题, 筛出 Top 15 候选
   ↓
3. 精检 (查准优先, 窄关键词块 + IPC)
   ↓
4. 读摘要 + 权要 1, 筛出 Top 5-10
   ↓
5. 全文精读
```

### "充分检索"的判断（sufficient search）

在以下任一条件满足时可停止：

1. **最后 20 篇命中无新的高相关度文献**
2. **已覆盖**：核心 IPC × 3 种以上同义词组合 × 中英双语
3. **非专利文献**：已查 Google Scholar + arXiv + CNKI
4. **时间预算**：达到既定上限

### 阅读量参考

| 检索目的 | 摘要阅读量 | 全文精读 |
|---|---|---|
| 专利可专利性评估 | 200-500 | 20-50 |
| FTO | 500-2000 | 50-100 |
| 无效检索 | 500+ | 无上限 |
| **初筛 (本 skill 场景)** | **50-150** | **5-15** |
| 查新检索 | 100-300 | 10-30 |

本 skill 是**初筛**，不是正式可专利性评估。目标是"足够"而不是"穷尽"。

## 8. 查询迭代的触发条件

Phase A 的 Screener 在以下情况应该做第二轮迭代：

1. 第一轮无相关命中 → 检查是否关键词过窄
2. 第一轮命中全是领域外的噪音 → 加 IPC 限定
3. 第一轮命中和本方案撞车太多 → 已发现破坏新颖性，直接走红灯

Phase B 的用户在以下情况可以扩展：

1. Top 20 里都是已知文献 → 检索式过窄或覆盖太旧
2. Top 20 里有大量噪音 → 需要加更严格的 IPC
3. 无相关命中 → 检查语义检索和 Prior Art Finder，两者都无则接受"Top 20 无相关命中"作为结论

## 9. 常见错误清单

| 错误 | 后果 | 缓解 |
|---|---|---|
| 仅中文检索 | 漏掉海外专利 / arXiv / IEEE | 强制双语检索 |
| 不做同族合并 | 重复读同一发明 | 默认开启去重 |
| 依赖单一 IPC | 漏掉跨领域 | 至少 2-4 个小组 |
| 关键词过窄 | 漏相关专利 | 至少 3 组块 |
| 忽略非专利 | 漏论文现有技术 | Scholar + arXiv 必查 |
| 截止日期设错 | 漏抵触申请 | 设今天, 单独查抵触 |
| 仅读摘要下结论 | 漏掉详细描述里的披露 | 精读文全文 |
| 上下位 / 术语替换当作区别 | 审查员驳回 | 做公知常识检查 |
| 只查新颖性不查创造性 | 三步法不过 | 两者都查 |
| 未查英文全文库 | CN 官方库弱项 | 配合 Google Patents |
