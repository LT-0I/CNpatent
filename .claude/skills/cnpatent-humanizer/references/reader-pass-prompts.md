# Reader-Pass Prompt Variants (分节 + 多人格)

> `reader_pass.py --mode prompt` 目前使用通用模板。本文档记录**分节 /
> 分子节 / 分人格**的 prompt 变体，可在需要更尖锐的读者视角时替换。
> 所有变体都遵循：
>
> 1. 零接触 audit 分数、Tier 命中、term-lock；
> 2. 每个 flag 必须带 `pattern_template + description + citations`；
> 3. 每个 citation.substring ≥ 8 字，逐字摘自原文；
> 4. 输出 JSON 数组；无 flag 时输出 `[]`；
> 5. 禁改写、禁内容建议。

所有 prompt 都以 `reader_pass.py` 的默认模板为基底，仅在 `persona /
section_name / 重点` 三个插槽上替换。

---

## 默认模板 (reader_pass.py 内嵌)

```
你正在审阅一份中国发明专利技术交底书的{section_name}部分。
作为资深专利工程师/审查员，请把下文当作一位同行代理人写的初稿阅读，
而不是作为格式检查器。

只关注修辞与结构层面，不改动内容，不提出技术建议。如果发现下列问题之一，
请用 JSON 数组输出 flag。每个 flag 必须同时包含:
  - pattern_template: 骨架模板引用 (如 "[状语前置]+[主语]+[否定/情态]+[动词]+[：机制]+[；数据]")
  - description: 一句中文描述问题
  - citations: 至少 2 个引用，每个含 item_ref (如 "（1）") 和 substring (从原文逐字摘取的一段，至少 8 字，用于事后校验)

需要识别的问题类型:
1. 兄弟条目修辞骨架重复 — 每条都按同一模板展开 (句首状语/主语/动词/分隔符/补语位置一致)
2. 信息密度均匀 — 每条都同时填满 {机制, 量化数据, 对比基线, caveat} 多个槽位
3. 论证完备度过高 — 每个主张都同时给出机制+数据+基线比较
4. 否定/情态并行 — 多条使用 [否定副词]+[情态]+[动词] 同一结构
5. 段首同类状语堆积 — 连续多条以「X下/X中/X之外」开头

如果没有发现骨架级问题，输出空数组 []。
不要评论内容对错、不要改写、不要建议，只识别骨架。
```

---

## §4 技术效果 (section=四, 子节 4c/发明效果)

**Persona**: 资深专利审查员

**追加重点**:
```
特别检查:
  A) 效果条目是否与本节之前的「发明目的」镜像对齐 (P_i ↔ E_i 一对一) —
     这是中文专利 §4 最强 AI 骨架信号
  B) 是否每条都带「相较 ... / 较 ... 下降」对比 → 对比槽位穷尽
  C) 是否每条都带「消融测试中 ... 下降 N 个百分点」量化数据 → 数据槽位穷尽
  D) 是否每条都以 [状语前置]+[否定副词]+[肯定动词] 三段链展开
  
注意: 单独一条罗列机制或单独一条罗列数据是合法的，问题只出现在
「每条都同时带多个槽位」的情况。
```

---

## §4 发明目的 (section=四, 子节 4a/发明目的)

**Persona**: 资深代理人

**追加重点**:
```
特别检查:
  A) 每条「目的」是否都以「为解决 ... 问题」或「针对 ... 难题」开头 → 骨架化
  B) 目的条数是否正好配合 §3 背景技术的问题条数 (3-to-3 镜像)
  C) 是否每条都把 motivation + mechanism 同时塞在一条里
  D) 段首是否连续用「其一 / 其二 / 其三」或连续「首先 / 其次 / 最后」
  
发明目的节天然带 motivation 槽位，这本身不是问题；
问题在于 motivation 与 mechanism / data 在同一条并列出现。
```

---

## §4 技术解决方案 (section=四, 子节 4b)

**Persona**: 技术员 / 实施工程师

**追加重点**:
```
特别检查:
  A) 步骤列表是否每条都以「通过 ... 实现 ...」同模板展开
  B) 是否每条都在分号后带「从而 / 进而 / 以便」连接副词
  C) 步骤数是否是完美偶数 (4/6/8) — 结构上可疑
  D) 每个步骤是否都带「方法 + 参数 + 模块」三位一体
  
解决方案节是步骤罗列，低信息密度是合法的；
重点盯「每条都塞满解释」。
```

---

## §6 具体实施方式 (section=六)

**Persona**: 资深代理人

**宽松模式 (critical)**:
```
注意: 子步骤 (a)(b)(c) 平行排列是中国专利 §6 的**合法写法**，
不视为 AI 骨架。本节 reader_pass 只在下列情况 flag:

  ONLY IF 每个 (a)(b)(c) 都同时带「机制 + 数据 + 对比 + caveat」多个
  论证槽位，而非单纯动作描述。

如果步骤只是「(a) 采集 X 数据; (b) 对 X 做 Y 处理; (c) 输出 Z」
这样的朴素动作描述，输出空数组 []。
```

## §3 背景技术 (section=三)

**Persona**: 资深专利审查员

**追加重点**:
```
特别检查:
  A) 问题枚举是否使用「其一 / 其二 / 其三」骨架
  B) 是否每条问题都以「现有技术 ... 存在 ... 问题」同句式
  C) 段末是否有「综上 / 因此 / 为此」之类的总结-过渡骨架
  D) 引用现有方案时是否每条都带「相较本发明 / 不同于」对比 — 反常，
     背景技术不应对比本发明

背景技术应当是「问题陈述 + 缺陷描述」，不应有 mechanism 或 data 槽位；
若看到 mechanism 进入背景技术条目，属 §3-§4 越界骨架。
```

## §5 附图说明 (section=五)

**Persona**: 代理人

**宽松模式**:
```
附图说明节为简短的一句话描述，天然平行：
「图 1 为本发明实施例中 X 的结构示意图;
 图 2 为本发明实施例中 Y 的流程示意图;...」

仅在下列情况 flag:
  A) 图说明带进「本发明/所述 ...」这种发明内容越界表达
  B) 条目带入量化数据 / 对比 / caveat

其他情况输出空数组 []。
```

---

## §1 发明名称 / §2 技术领域 (section=一 / 二)

**Persona**: 代理人

**宽松模式**:
```
§1 通常只有一行，几乎永远输出 []。
§2 通常 1-2 句，flag 条件:
  A) 使用推销语 (如「广阔的应用前景」「极大的应用价值」)
  B) 同段内相同骨架 ≥3 次重复
其他情况输出空数组 []。
```

## all (完整文档跨节扫描)

**Persona**: 资深专利工程师

**追加重点**:
```
全文扫描时特别检查:
  A) §3 背景问题枚举条数 与 §4.1 发明目的条数 与 §4.3 技术效果条数
     是否精确对齐 → 跨节镜像骨架
  B) 同一骨架 (如「状语前置 + 否定 + 情态」) 是否在不同章节重复出现
  C) 是否存在「发明目的 → 技术解决方案 → 技术效果」三节套同一三元论证
     骨架 (problem + mechanism + data)

跨节镜像是 reader_pass 独有检测能力：节内检测器 (skeleton_sim,
argumentation_slots) 只看单节兄弟；镜像需要跨章节视角。
```

## 人格切换约定

| section | persona 默认 | 变体可选 |
|---|---|---|
| 一 | 代理人 | — |
| 二 | 代理人 | 资深审查员 (尖锐化) |
| 三 | 资深审查员 | 代理人 (放松) |
| 四 (目的) | 代理人 | 资深审查员 (严格) |
| 四 (方案) | 技术员 | 代理人 |
| 四 (效果) | 资深审查员 | 资深工程师 |
| 五 | 代理人 | — |
| 六 | 代理人 (宽松) | — |
| all | 资深工程师 | 资深审查员 |

人格差异只影响 `description` 语气，不影响 `pattern_template` 分类。

## 调用 prompt 变体 (未来扩展)

目前 `reader_pass.py --mode prompt` 只使用单一内嵌模板。若需按 section
切换变体，在 `build_prompt()` 添加：

```python
PROMPT_VARIANTS = {
    '四_effect': PROMPT_TEMPLATE + "\n追加重点: ...",
    '四_purpose': PROMPT_TEMPLATE + "\n追加重点: ...",
    '六': PROMPT_TEMPLATE + "\n宽松模式: ...",
}
```

并扩展 CLI `--section 四_effect` 支持子节键。本版本 P0 未实装；子节
追加规则作为**文档合同**记录在本文件，orchestrator 在调用 reader 时
可以手动拼接追加段到 prompt 尾部。
