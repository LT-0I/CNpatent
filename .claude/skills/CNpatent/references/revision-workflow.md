# CNpatent 修改工作流协议（Revision Workflow）

本文件定义 DOCX 交付后的修改流程。由 orchestrator 在收到用户修改意见时读取并执行。

---

## R-1：接收用户修改意见

接受两种输入格式：

**A. 自然语言（默认）**：用户直接描述修改意图，orchestrator 解析并分级。例：
- "学习率改成 0.001" → 微调
- "步骤 3 要加具体的 K 近邻参数说明" → 段落
- "技术方案部分重新组织，先介绍特征提取再介绍匹配" → 章节
- "整体换思路，不做点云补全了，改成点云分割" → 大纲

**B. 结构化模板（复杂修改时提供给用户）**：

```
修改级别（微调/段落/章节/大纲）：
涉及章节（如 4b_solution.md）：
修改意见：
保留内容（如不需要保留则留空）：
```

---

## R-2：4 级分类标准

### 微调（Micro）

**判定条件（全部满足）**：
- 不改变任何技术结论或论证逻辑
- 涉及修改：数值替换、措辞替换、格式规范（标点/编号/全半角）、术语统一

**示例**：
- "学习率从 0.01 改成 0.001"
- "把'点云修复'全文改成'点云补全'"
- "步骤（3）里的半角括号改成全角"

**执行**：Orchestrator 直接 `Edit` 对应 `sections/*.md`，无需调用 Writer。

---

### 段落（Paragraph）

**判定条件**：
- 语义改变（增加/删除/调整技术内容）
- 涉及范围：1-3 个自然段
- 不影响其他章节的对应关系

**示例**：
- "步骤（3）里要加入 K 近邻参数的具体说明，K 取 15"
- "背景技术第二段对现有方法的批评要更具体，加上计算复杂度 O(N^2)"

**执行**：路由到对应 Writer，提供修改上下文（见 R-4），Writer 只修改指定段落，其余内容保留。

---

### 章节（Section）

**判定条件**：
- 语义改变影响整节结构
- 需要重写或重组整个章节

**示例**：
- "技术解决方案部分重新组织，步骤顺序调整：先特征提取，再位置编码，再匹配"
- "具体实施方式需要增加一个实施例变体（小样本场景）"

**执行**：路由到对应 Writer，提供修改上下文（见 R-4），Writer 重写整节。

---

### 大纲（Outline）

**判定条件（满足任一）**：
- 技术路线发生变化（核心方法替换）
- 三方对应结构改变（优势数量/名称变化）
- 修改影响 2 个以上章节的对应关系

**示例**：
- "整体换思路，不做点云补全了，改成点云分割"
- "发明目的从 3 条优势改成 4 条，新增一条关于实时性的优势"

**执行**：

1. **警告**（必须，用户确认前不执行）：

```
⚠️ 您的修改属于大纲级变更，涉及技术路线或三方对应结构。

影响评估：
- 需重写章节：[列出受影响的 sections/*.md]
- 新颖性风险：[如技术方案实质变化，原 noveltycheck 验证结论可能失效]

建议：如技术方案发生实质变化，请先重跑 CNpatent-noveltycheck 做新颖性核验。

确认继续修改？（是/否）
```

2. 用户确认后：修改 `01_outline.md` → 重启 Phase 1（全部 4 个 Writer 重新并行生成）→ Phase 2 → Phase 3 → Phase 4

---

## R-3：路由决策树

```
收到修改意见
    |
    +-- 无语义变化（数值/措辞/格式）
    |       -> 微调：Orchestrator 直接 Edit
    |
    +-- 1-3 段语义变化，不影响其他章节
    |       -> 段落：路由对应 Writer + 修改上下文
    |
    +-- 整节结构调整
    |       -> 章节：路由对应 Writer + 修改上下文（全节重写）
    |
    +-- 技术路线变化 / 三方对应结构变化 / 多节联动
            -> 大纲：警告 -> 用户确认 -> 修改 01_outline.md -> Phase 1 重启
```

**章节到 Writer 的映射**：

| sections 文件 | 对应 Writer |
|---|---|
| 1_name.md, 2_field.md, 3_background.md | Writer-A |
| 4a_purpose.md, 4b_solution.md, 4c_effect.md | Writer-B |
| 5_figures.md | Writer-C |
| 6_implementation.md（前半部分） | Writer-C |
| 6_implementation.md（后半部分） | Writer-D |

> 对于 6_implementation.md，orchestrator 根据修改意见涉及的步骤编号判断路由到 Writer-C 还是 Writer-D，或两者都路由。

---

## R-4：Writer 重调用上下文模板

段落级和章节级修改时，orchestrator 构建以下上下文包后再调用 Writer：

```
你是 [Writer-X]，现在执行修改任务（不是初次生成）。

## 原始大纲片段
[从 01_outline.md 中提取与本节相关的部分]

## 当前章节内容（你上一次的输出）
[sections/[target].md 的完整现有内容]

## 用户修改意见（原文）
[用户的原始意见，逐字引用]

## 修改范围
[段落级：指定段落位置/编号]
[章节级：整节重写]
保留范围之外的内容不得改动。

## 术语锁定表
[从 01_outline.md 提取，与原始生成时相同]

## 写作规范
[注入 writing-rules.md 的禁用词表和句式检测规则，与初次生成一致]
```

**关键约束**：
- Writer 必须能看到"当前章节内容"，避免重新引入已修复的问题
- "保留范围之外的内容不得改动"必须明确，防止 Writer 顺手修改无关段落
- 术语锁定表与初次生成时完全相同，防止漂移
- 段落级修改时，Writer 输出完整修改后的文件（不只输出修改段），由 orchestrator 写回

---

## R-5：Reviewer 修改模式（轮次重置）

修改工作流完成后进入 Phase 2 时：

- **轮次计数器重置为 0**：与初次生成一样，最多 2 轮 Reviewer → Writer → Reviewer
- **Reviewer 收到额外上下文**：
  - "本次为修改后审查，用户修改意见：[原文]"
  - "本次修改级别：[微调/段落/章节/大纲]"
  - "已修改的文件：[列表]"
- **审查重点**：Reviewer 应优先检查修改范围内的一致性（三方对应、术语、图引用），同时执行完整 Rubric-A/B/C

Reviewer 在输出总结时注明："本次为修改后审查（轮次 1/2）"。

---

## R-6：反模式清单（Anti-Patterns）

以下行为禁止：

| 反模式 | 说明 |
|---|---|
| 把段落级意见升级为大纲级 | 除非修改确实影响三方对应或技术路线，不得触发大纲警告 |
| 跳过 Reviewer 直接出 DOCX | 任何 Writer 修改后都必须经过 Phase 2 Reviewer |
| Writer 修改范围外的内容 | 段落级修改时 Writer 只能动指定段落，其余内容原样保留 |
| Reviewer 自己修复语义问题 | 与初次审查一样，语义/结构问题必须退回 Writer |
| 大纲级未警告直接执行 | 大纲级变更必须先警告并等待用户明确确认 |
| 忽略三方对应 | 任何涉及优势/效果/背景局限数量或标题的修改都必须维持三方对应 N=N=N 且标题一字不差 |
